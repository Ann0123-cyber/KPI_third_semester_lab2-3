# ADR 001: Rich Domain Model для Task

**Статус:** Прийнято  
**Дата:** 2025

## Контекст

При розробці доменного шару необхідно обрати між двома підходами:
- **Anemic Domain Model** — моделі є пасивними контейнерами даних, вся логіка в сервісах/use cases.
- **Rich Domain Model** — моделі містять поведінку та самостійно захищають свої інваріанти.

## Рішення

Обрано **Rich Domain Model** для сутності `Task`. Для `User` та `Project` використано анемічний підхід, оскільки їхня поведінка тривіальна.

## Обґрунтування

Сутність `Task` має нетривіальний інваріант: переходи між статусами (`todo → in_progress → done`) суворо регламентовані. Цей інваріант логічно належить самому об'єкту, а не зовнішньому сервісу.

У Rich підході:
```python
task.transition_to(TaskStatus.DONE)  # сам знає, що можна, а що ні
```

В анемічному підході логіку довелося б дублювати в кожному use case, який змінює статус задачі, що збільшує ризик пропустити перевірку.

## Переваги

- Інваріант `transition_to()` неможливо обійти — він завжди виконується при зміні статусу.
- Доменна логіка тестується без use cases, без БД — чистий unit-тест над `Task`.
- Читабельний код: `task.transition_to(...)` чіткіше виражає намір, ніж `if status not in allowed: raise`.

## Недоліки

- Модель `Task` стає більшою за анемічний аналог.
- При великій кількості поведінки клас може роздутися — тоді варто виділяти доменні сервіси.

## Альтернатива

Якщо б бізнес-логіка була переважно процедурною (batch-обробка, складні запити) — Anemic був би доречніший.
