# Use Cases — Task Manager API

## UC-01: Реєстрація користувача

**Актор:** Незареєстрований відвідувач  
**Передумови:** Жодних (відкритий endpoint)

**Основний сценарій:**
1. Користувач надсилає `POST /auth/register` з `email`, `username`, `password`
2. Система перевіряє формат email, довжину пароля (≥ 8 символів), формат username
3. Система перевіряє унікальність email та username
4. Система створює запис користувача з хешованим паролем
5. Повертає дані користувача (201 Created)

**Альтернативні сценарії / помилки:**
- Email або username вже зайняті → 409 Conflict
- Невалідний email, короткий пароль, недопустимий username → 422 Unprocessable Entity

---

## UC-02: Вхід у систему

**Актор:** Зареєстрований користувач  
**Передумови:** Обліковий запис існує

**Основний сценарій:**
1. Користувач надсилає `POST /auth/login` з `email` та `password`
2. Система верифікує пароль через bcrypt
3. Генерується JWT-токен (час дії — 60 хвилин)
4. Повертається токен (200 OK)

**Альтернативні сценарії / помилки:**
- Невірний email або пароль → 401 Unauthorized

---

## UC-03: Перегляд власного профілю

**Актор:** Автентифікований користувач  
**Передумови:** Дійсний JWT-токен у заголовку `Authorization: Bearer <token>`

**Основний сценарій:**
1. Користувач надсилає `GET /auth/me`
2. Система декодує токен, знаходить користувача
3. Повертає дані профілю (200 OK)

**Альтернативні сценарії / помилки:**
- Токен відсутній → 403 Forbidden
- Токен прострочений або невалідний → 401 Unauthorized

---

## UC-04: Створення проекту

**Актор:** Автентифікований користувач  
**Передумови:** Користувач авторизований

**Основний сценарій:**
1. Користувач надсилає `POST /projects/` з `name` (та опційним `description`)
2. Система перевіряє: назва не порожня, не перевищує 128 символів
3. Система перевіряє унікальність назви серед проектів цього користувача
4. Проект створюється (201 Created)

**Альтернативні сценарії / помилки:**
- Проект з такою назвою вже існує → 409 Conflict
- Порожня або задовга назва → 422 Unprocessable Entity

---

## UC-05: Отримання списку проектів

**Актор:** Автентифікований користувач  
**Передумови:** Користувач авторизований

**Основний сценарій:**
1. Користувач надсилає `GET /projects/`
2. Повертаються лише його власні проекти (200 OK)

**Примітка:** Проекти інших користувачів не видимі.

---

## UC-06: Оновлення проекту

**Актор:** Автентифікований власник проекту  
**Передумови:** Проект існує і належить користувачу

**Основний сценарій:**
1. Користувач надсилає `PATCH /projects/{id}` з оновленими полями
2. Система перевіряє інваріанти (назва не порожня, нема конфлікту імен)
3. Дані оновлюються (200 OK)

**Альтернативні сценарії / помилки:**
- Проект не знайдено або чужий → 404 Not Found
- Конфлікт назви → 409 Conflict

---

## UC-07: Видалення проекту

**Актор:** Автентифікований власник проекту  
**Передумови:** Проект існує і належить користувачу

**Основний сценарій:**
1. Користувач надсилає `DELETE /projects/{id}`
2. Проект та всі пов'язані задачі видаляються (cascade) (204 No Content)

**Альтернативні сценарії / помилки:**
- Проект не знайдено → 404 Not Found

---

## UC-08: Створення задачі

**Актор:** Автентифікований користувач (власник проекту)  
**Передумови:** Проект існує і належить користувачу

**Основний сценарій:**
1. Користувач надсилає `POST /projects/{project_id}/tasks/` з `title`, `priority`, `due_date`, `assignee_id`
2. Система перевіряє:
   - Назва задачі не порожня, ≤ 256 символів
   - `due_date` — у майбутньому (якщо вказана)
   - `assignee_id` — існуючий користувач (якщо вказаний)
3. Задача створюється зі статусом `todo` (201 Created)

**Альтернативні сценарії / помилки:**
- Проект не знайдено → 404 Not Found
- Assignee не знайдено → 404 Not Found
- Дата в минулому або порожня назва → 422 Unprocessable Entity

---

## UC-09: Оновлення задачі / зміна статусу

**Актор:** Автентифікований користувач (власник проекту)  
**Передумови:** Задача існує у проекті користувача

**Основний сценарій:**
1. Користувач надсилає `PATCH /projects/{project_id}/tasks/{task_id}`
2. Система перевіряє інваріанти полів
3. При зміні статусу перевіряється допустимість переходу:

| Поточний статус | Допустимі переходи          |
|----------------|-----------------------------|
| `todo`         | `in_progress`, `cancelled`  |
| `in_progress`  | `done`, `todo`, `cancelled` |
| `done`         | `todo`                      |
| `cancelled`    | `todo`                      |

4. Задача оновлюється (200 OK)

**Альтернативні сценарії / помилки:**
- Недопустимий перехід статусу → 400 Bad Request
- Задача або проект не знайдені → 404 Not Found

---

**Основний сценарій:**

**Альтернативні сценарії / помилки:**
