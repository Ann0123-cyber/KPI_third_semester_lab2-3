# Аналіз: Лабораторна 3 — CQS

## Що змінилося порівняно з лабораторною 2

У лабораторній 2 application шар складався з use cases — кожен клас `CreateProjectUseCase`, `UpdateProjectUseCase` тощо міг і читати, і писати. `GetProjectUseCase` читав через репозиторій, повертав доменну модель, яку потім маппер перетворював у DTO. Різниці між "операцією читання" і "операцією запису" в коді не було — обидва йшли через один і той самий механізм.

У лабораторній 3 application шар явно розділений на дві частини: `commands/` + `handlers/command_handlers/` для операцій запису, і `queries/` + `handlers/query_handlers/` для читання. З'явились `read_models/` — окремі DTO оптимізовані під потреби UI. Query Handlers читають напряму з БД через SQLAlchemy, минаючи доменний шар і репозиторії. Command Handlers проходять через домен, фабрики та репозиторії — як і раніше.

## Переваги CQS

**Читання оптимізоване незалежно від запису.** У лаб 2 `GetProjectUseCase` повертав доменну `Project`, яка не мала поля `task_count`. Щоб додати кількість задач до відповіді, довелося б або змінювати доменну модель (порушення), або робити додатковий запит у роутері (жахливо). У лаб 3 `ListProjectsHandler` робить один JOIN запит і повертає `ProjectReadModel` з полем `task_count` — без жодних змін у домені. Так само `TaskReadModel` містить `assignee_username` — денормалізоване поле, яке UI потребує, але домену воно не потрібне.

**Команди і запити тестуються по-різному і незалежно.** Command Handler тестується unit-тестом з fake репозиторієм — швидко, без БД, перевіряє бізнес-інваріанти. Query Handler тестується integration-тестом через HTTP — перевіряє що SQL правильний, JOIN працює, поля у відповіді присутні. Кожен тип тесту перевіряє саме те, що важливо для свого типу операцій.

**Команди явно не повертають дані.** `CreateProjectHandler.handle()` повертає лише `int` (ID). `UpdateProjectHandler.handle()` повертає `None`. Це робить намір очевидним: команда змінює стан, але не читає його. Якщо треба отримати оновлені дані — роутер після команди викликає Query Handler.

## Недоліки CQS

**Більше файлів і більше навігації.** У лаб 2 `UpdateProjectUseCase` — один файл. У лаб 3: `UpdateProjectCommand` + `UpdateProjectHandler` — два файли, плюс для читання після оновлення роутер викликає `GetProjectHandler`. Щоб прослідкувати один `PATCH /projects/{id}`, треба відкрити 4-5 файлів.

**Query Handlers залежать від інфраструктури напряму.** Вони отримують `Session` і пишуть SQLAlchemy-запити. Це свідоме порушення ізоляції заради оптимізації — але тестувати їх unit-тестами без БД не вийде. Integration-тести для запитів обов'язкові.

**POST повертає дані через додатковий запит.** Після `CreateProjectHandler` роутер одразу викликає `GetProjectHandler` щоб повернути повну Read Model клієнту. Це два звернення до БД замість одного. В лаб 2 use case повертав доменну модель яку одразу маппували — один раз.

## Чим Command/Query Handler відрізняється від Service

`UpdateProjectUseCase` у лаб 2 робив все сам: читав, валідував, оновлював, повертав результат. Це один клас з однією відповідальністю в теорії, але з трьома в практиці. При зміні формату відповіді треба було чіпати use case навіть якщо бізнес-логіка не змінювалась.

`UpdateProjectHandler` у лаб 3 лише змінює стан і повертає `None`. Формат відповіді — відповідальність `GetProjectHandler`. Змінився формат відповіді — міняємо Query Handler і Read Model, Command Handler не чіпаємо. Змінилась бізнес-логіка — міняємо Command Handler, Query Handler не чіпаємо.

## Як CQS впливає на розширюваність

Додати нову команду — створити два файли: `XxxCommand` і `XxxHandler`. Жодний існуючий код не змінюється. Додати нову query з іншим набором полів — так само: новий `XxxQuery`, новий `XxxHandler`, новий `XxxReadModel`. Можна мати кілька Read Models для одної сутності: `ProjectReadModel` для списку з `task_count`, і `ProjectDetailReadModel` для деталей з повним списком задач — і вони не заважають одна одній.

У лаб 2 щоб додати нове поле до відповіді довелося б: змінити доменну модель або маппер, змінити use case, змінити схему. Чотири місця. У лаб 3 — лише Read Model і Query Handler.

## Чи відрізняється структура даних Query від доменної моделі

Так, і це принципово важливо. Доменна модель `Project` має `id`, `name`, `description`, `owner_id`, `created_at` — лише те, що потрібно для бізнес-логіки. `ProjectReadModel` додає `task_count` — денормалізоване поле яке зручне для UI але не є частиною бізнес-правил. `TaskReadModel` додає `assignee_username` — щоб UI не робив окремий запит за ім'ям виконавця.

Якби Query повертав доменну модель, довелося б або засмічувати домен полями для UI, або робити додаткові запити в роутері. Read Model — це контракт між backend і frontend, оптимізований під конкретний екран, незалежний від доменних правил.
