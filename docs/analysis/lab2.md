# Аналіз: Лабораторна 2 vs Лабораторна 1

## Що змінилося в структурі проекту

У лабораторній 1 проект мав плоску структуру: `models/`, `schemas/`, `services/`, `routers/`. Бізнес-логіка, ORM, HTTP та валідація перемішувалися — сервіс безпосередньо викликав `db.query()`, а доменні правила жили поряд із SQLAlchemy-об'єктами. Не було чіткої межі між тим, що є «правилом домену», а що — «деталлю реалізації».

У лабораторній 2 з'явилися 4 чіткі шари: `domain/`, `application/`, `infrastructure/`, `presentation/`. Домен став ізольованим — жодного імпорту SQLAlchemy, FastAPI чи bcrypt. ORM-моделі (`UserORM`, `ProjectORM`, `TaskORM`) повністю відокремлені від доменних (`User`, `Project`, `Task`). Між ними з'явився шар маперів. Інтерфейси репозиторіїв визначені в домені, реалізовані в інфраструктурі — це і є DIP на практиці.

## Переваги розділення на шари

Найголовніша перевага — **тестованість**. У лаб 1 будь-який unit-тест, що торкався сервісу, потребував бази даних. Тепер доменні тести (`test_task_model.py`, `test_factories.py`) та тести use cases (`test_project_use_cases.py`) запускаються без жодної інфраструктури — лише чисті Python-об'єкти та fake-репозиторії. Це пришвидшує запуск тестів і знижує крихкість тест-сюїту.

Друга перевага — **явність залежностей**. Коли `CreateProjectUseCase` приймає `IProjectRepository` замість конкретного `SqlProjectRepository`, одразу видно: use case не залежить від бази даних. Це зменшує когнітивне навантаження і спрощує онбординг нових розробників.

## Недоліки та ускладнення

Кількість файлів зросла приблизно втричі. З'явився маппінг (3 файли маперів), який є суто технічним кодом без бізнес-цінності, але його потрібно підтримувати синхронно з обома сторонами (ORM та доменна модель). При зміні поля в доменній моделі треба оновити і ORM, і маппер, і тест — три точки замість однієї.

Навігація по проекту ускладнилась: щоб прослідкувати шлях запиту від HTTP до бази, треба пройти через роутер → use case → фабрику → репозиторій-інтерфейс → SQL-реалізацію → маппер → ORM. У лаб 1 це були 2 файли, тепер — 6.

## Наскільки простіше тепер змінити БД або фреймворк

**Зміна БД:** для переходу з PostgreSQL на MongoDB треба переписати лише `infrastructure/repositories/` та `infrastructure/orm/` — 6 файлів. Доменні моделі, use cases, фабрики, роутери та тести залишаються незмінними. У лаб 1 потрібно було б торкнутися сервісів, моделей і тестів.

**Зміна фреймворку:** FastAPI присутній лише в `presentation/` та `app/main.py`. Увесь домен і application шар не знають про FastAPI. Теоретично можна замінити FastAPI на Flask, змінивши лише роутери та схеми — бізнес-логіка не зміниться.

## Чому обрано Rich Domain Model

Детальне обґрунтування — в ADR 001. Коротко: сутність `Task` має нетривіальний інваріант (переходи між статусами), який логічно належить самому об'єкту. Метод `task.transition_to(new_status)` неможливо випадково обійти — він завжди виконується при зміні статусу, незалежно від того, який use case ініціює зміну. В анемічному підході цю перевірку довелося б дублювати в кожному use case.
